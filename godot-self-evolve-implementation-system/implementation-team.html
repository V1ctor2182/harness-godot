Section 01

Agent 角色 — AI Implementation Team

6 个专职 Agent 在 Docker 容器中自主协作。安全来自流程守护，不来自能力限制。

🎯

Orchestrator

编排者
读取 Milestone + PRD + Feature Room spec，将目标分解为 3-7 个 Task，输出 JSON 计划。
PLAN 阶段超时 5min$3/run
  • 分析最近 3 个 cycle 的 retrospective 检测重复模式
  • 最小化 estimatedFiles 重叠，防止并行合并冲突
  • 为每个 task 指定 testScenarios（L2/L3 测试路径）
  • 标注 featureRooms + crossDomainRisks
  • 对 .tscn 和 data/global/ 施加互斥约束
  • 输出 contextFeedback: useful / missing / unnecessary
⌨️

Coder ×N

开发者（可并行）
读取 Task Spec + PRD + Room Spec，实现功能、写 L1 单元测试、生成占位素材、提交 PR。
IMPLEMENT 阶段超时 10min$5/run
  • 从 BASE_BRANCH 创建 task 分支
  • 资源引用必须通过 AssetManager（禁止裸 load()）
  • 新资源同步注册到 registry/*.json + placeholder_manifest
  • L1 GUT 测试自己跑通后才提 PR
  • PR body 包含 Decisions Made + Constraints Discovered
  • 每次 commit 立即 push（防容器崩溃丢失代码）
  • 修复循环中在同一分支继续，带累积错误上下文
🧪

Tester

测试者
执行 L2-L4 四层测试，发现问题生成 fix-task，驱动修复闭环。
TEST 阶段超时 10min$3/run
  • L2: Headless 集成测试 + 性能基线（FPS≥60, 节点≤500）+ 资源校验
  • L3: MCP Pro 截图 + AI 视觉分析（异步，不阻塞 REVIEW）
  • L4: PRD 公式/常量/数据合规检查（可与 L2 并行）
  • 失败时生成 fix-task（含失败断言 + 截图 + 建议方向）
  • 快速失败: L1 不过→不跑 L2, L2 不过→不跑 L3
  • 占位素材只验证"存在且可见"，不评判美术质量
🔍

Reviewer

审查者
审查代码质量、PRD 合规、架构一致性、资源管线合规，输出 approved 或 changes-requested。
REVIEW 阶段超时 5min$2/run
  • 7 项审查清单: 代码质量/PRD 合规/架构一致/测试覆盖/数据同步/文档完整/资源管线
  • 按 task type 应用不同标准（feature/bug/refactor/test）
  • 交叉校验 Coder 的 contextFeedback 是否与代码一致
  • changes-requested 计入重试计数（REVIEW 阶段上限 2 次）
🔀

Integrator

集成者
按依赖顺序合并 PR、运行全量回归、更新 Milestone 文档、处理冲突和回滚。
INTEGRATE 阶段动态超时 5-30min
  • 正常路径可降级为脚本（integrate.sh），只在冲突/回归失败时启动 AI
  • dry-run merge 检测冲突 → .tscn 冲突用 Godot headless 解决
  • 全量回归: L1 全量 + L2 受影响模块 + L3 关键路径
  • 回归失败: 先尝试 hotfix（2 次），再 git revert
  • 同一 Milestone 内 >2 次回滚 → 系统暂停，人类架构审查
📚

Curator

策展者
知识沉淀 + 演化提案。从 PR 中提取 decision/constraint 写入 Feature Room，识别改进机会写入 evolution-inbox。
SEDIMENT + REFLECT按需启动
  • SEDIMENT: 读 PR body + diff → 提取 decision/constraint/context → 写入 Room spec.md
  • 置信度标注: high（PR 明确声明）/ medium（diff 推断）/ low（间接推断，标记[待确认]）
  • 演化提案: 重复失败 → Layer 1 checklist / 系统性问题 → Layer 2 分析框架
  • REFLECT: 审查 agents/*.md 每条规则 → KEEP/MERGE/OUTDATED/CONFLICT/UNTESTED/FROZEN
  • 合并冗余 decision，不删除有效知识

🎯 PLAN — Orchestrator 编排

输入: Milestone 文档 + PRD + Feature Room spec + 最近 3 个 retrospective + known-issues

输出: JSON 任务计划（3-7 个 task），每个 task 含 acceptanceCriteria / prdRefs / testScenarios / estimatedFiles / featureRooms / crossDomainRisks

关键规则:

  • 最小化文件重叠（防并行合并冲突）
  • .tscn 和 data/global/ 施加互斥约束
  • 同一 Room 的并行 task 必须有不重叠的 estimatedFiles
  • plan-validator 做结构验证（字段完整性、无循环依赖）

⌨️ IMPLEMENT — Coder 实现

上下文注入: Task Spec + PRD 全文 + 目标 Room spec + 父 Epic Room + Project Room + depends_on Room 摘要 + conventions + registry/*.json

工作流: 读需求 → 探索代码 → 创建分支 → 生成占位素材 → 实现功能 → 写 L1 测试 → 跑 GUT → 通过 → 提 PR

🧪 TEST — 四层测试 + 修复闭环

快速失败原则: L1 不过→不跑 L2 → 不跑 L3。L4 可与 L2/L3 并行。

L3 异步: L1+L2 通过后 task 进入 REVIEW，L3 后台排队。

🔍 REVIEW — Reviewer 审查

7 项审查清单:

  • 代码质量 — GDScript 规范、命名约定、信号使用
  • PRD 合规 — acceptanceCriteria 全部有验证证据
  • 架构一致性 — Autoload / 信号驱动 / 场景树
  • 测试覆盖 — 按 testRequirements 覆盖所有层级
  • 数据同步 — JSON / data.md / project.godot 一致
  • 文档完整 — PR body Scene/Signal/Data Changes 已填写
  • 资源管线 — 注册完整、无裸 load()、占位清单已更新

🔀 INTEGRATE + RETROSPECT

合并: dry-run merge → 冲突检测 → 按 blockedBy 顺序合并 → 全量回归

回归策略:

  • 快速回归（每 cycle）: L1 全量 + L2 受影响模块 + L2 核心路径 — 目标 5 分钟
  • 完整回归（Milestone 边界）: L1 + L2 + L3 全量 — 不限时间

回滚: 回归失败 → hotfix（2 次）→ git revert → 人类审查。回滚时 Curator 标记 decision 为 (REVERTED)

📚 SEDIMENT — 知识沉淀

Curator 读取本 cycle 所有 PR body + diff:

  • 提取 decision → 写入 Room spec.md（附 cycle 引用和置信度）
  • 提取 constraint → 写入 Room spec.md（必须可验证）
  • 提取 context → 写入 Room spec.md（设计意图、失败教训）
  • 识别重复失败 → 写入 evolution-inbox/layer1-*.md
  • 识别系统性问题 → 写入 evolution-inbox/layer2-*.md
  • 更新 room.yaml lifecycle / code_refs / _tree.yaml

🪞 REFLECT — Prompt 防膨胀

触发条件（任一）: Milestone 完成 / agents/*.md 行数增长 ≥30% / inbox ≥10 条 / 人类手动

独立预算: $10/次，不占用 cycle 预算。每次最多审查 3 个 agents/*.md，处理 ≤15 条提案。

详见下方 REFLECT 机制 专节。

4 种 Spec 类型

intent — 意图

这个功能要做什么。从 PRD 提取。

例: "用户可以在实验室选僵尸+催化剂启动突变"

decision — 决策

关键技术/设计选择 + 理由。implement 过程中 Curator 沉淀。

例: "融合算法：70% 合计属性 + 20% bonus (M8-C1)"

constraint — 约束

不做什么 / 边界条件。必须可验证。

例: "衰变到 50% 停止不会死亡"

context — 背景

设计动机、失败教训、被否决方案。帮助理解 why。

例: "衰变系统的目的是制造管理压力"

Agent 如何使用 Room
Orchestrator 读 — 了解已有 decision，避免冲突
Coder 读 — 获取功能域上下文和设计意图
Tester 读 — 用 constraint 做验证依据
Reviewer 读 — 检查代码是否违反 constraint
Curator 读+写 — 每个 cycle 沉淀新知识

永远阻断，不受信任等级影响

🔮 CLUSTER — 聚类分析

  • 按 target_agent 分组 evolution-inbox 中所有提案
  • 同一 agent 的提案按主题聚类（如 "retry 处理" 相关归为一组）
  • 互补提案合并为一条更完整的规则
  • 矛盾提案标记为 CONFLICT → 列出冲突对提交人类裁决
  • 优先级: 3+ cycle 证据 > 2 cycle > 1 cycle; Layer 1 优先于 Layer 2

🔎 AUDIT — 逐行盘点

Curator 逐行审查目标 agents/*.md，标注每条规则:

KEEP仍有效 + 最近 3 cycle 被引用 MERGE与另一条重叠，可合并 OUTDATED针对的问题已不存在 CONFLICT与另一条规则矛盾 UNTESTED添加后从未被 useful 引用 → 进入观察期 FROZENdomain-scoped 规则近期无该域 task → 豁免，不计入膨胀

🔧 CONSOLIDATE — 整理

  • 合并 [MERGE] 条目 — 两条变一条，保留两者的 cycle 引用
  • 删除 [OUTDATED] 条目 — 已被架构/代码变更解决
  • 解决 [CONFLICT] — 保留更新的那条，或重写为无歧义版本
  • [UNTESTED] → 标记"观察中"（再给 3 cycle），域规则冻结超 2 个 Milestone 且仍未引用才删除

📥 INTEGRATE INBOX — 整合待处理提案

  • Layer 1 + Layer 2 提案 → 合并到 agents/*.md 对应段落
  • Layer 3 提案 → 提交 Human Gate 审批（不自动合并）
  • 每次最多处理 15 条提案，超出保留到下次 REFLECT
