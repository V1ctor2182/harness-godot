# 08 — Project Setup

How harness-system connects to a game project. The `.ludus/` contract, the setup flow, and the first-cycle bootstrap.

## P1. Overview

Ludus-system is a project-agnostic AI engineering team. It needs a **target project** to work on — the game repo. The connection between harness and project is defined by two things:

1. **`PROJECT_REPO_LOCAL_PATH` env var** — absolute path to a local checkout of the game repo
2. **`.ludus/project.yaml`** — the project's identity card inside that repo

When both are set, Ludus reads the project config at startup, seeds project Feature Rooms, and enables cycle creation. Without them, Ludus boots in "no project" mode: dashboard shows an empty state, cycle creation returns 409.

## P2. `.ludus/` Directory

```
game-repo/
├── prd/                            ← product docs (game-owned)
├── scripts/, scenes/, tests/...    ← game source code
└── .ludus/
    ├── project.yaml                ← identity + tech stack + conventions
    └── rooms/                      ← Feature Rooms (Curator writes, agents read)
```

That's it. Two entries.

**What does NOT live in `.ludus/`:**

| What | Why not | Where instead |
|---|---|---|
| Agent prompts | Ludus owns prompts; project provides context via `project.yaml` | `harness-system/agents/*.md` |
| Knowledge files | Replaced by Feature Room specs | `.ludus/rooms/` |
| Milestones | Generated by Orchestrator from PRD, confirmed by human, stored in Mongo | Dashboard management UI |
| Asset manifests | Derived from filesystem scan + PRD | `/api/assets` scanner at runtime |
| PRD documents | Product docs, not harness config | `game-repo/prd/` (root level) |

## P3. `project.yaml` Full Schema

```yaml
project:
  # ── Required ──────────────────────────────────────────
  id: "zombie-farm"             # unique, used as DB namespace
  name: "Zombie Farm"           # display name in dashboard

  # ── Optional ──────────────────────────────────────────
  description: |                # injected into all agent prompts
    A zombie farming simulation with xianxia cultivation elements.

  stack:                        # auto-detected by POST /api/project/init
    engine: "Godot"             # Godot | Rust | Node | Go | Python | Unknown
    engine_version: "4.6.1"     # free-form string
    language: "GDScript"        # primary language
    test_runner: "GUT"          # testing framework name
    os: "linux-amd64"           # container OS hint

  paths:                        # key dirs relative to repo root
    source: "scripts/"
    tests: "tests/"
    data: "resources/"

  conventions: |                # free-text, appended to all agent system prompts
    Use static typing (:=, -> void) throughout.
    File names use snake_case.
    Signals use past tense (health_changed, zombie_spawned).
    All nodes have class_name declarations.

  prd_path: "prd/"              # where Orchestrator finds PRD docs

  constants:                    # project-specific overrides (key-value)
    gut_timeout_ms: 180000
    tile_size: 64

  test_layers:                  # test layer definitions (dynamic, not hardcoded)
    - id: unit
      name: "Unit Tests"
      runner: "gut"
    - id: integration
      name: "Integration Tests"
      runner: "gut"
```

**Auto-detected fields** (by `POST /api/project/init`):

| Marker file | Detected engine | Detected language |
|---|---|---|
| `project.godot` | Godot | GDScript |
| `Cargo.toml` | Rust | Rust |
| `package.json` | Node | TypeScript |
| `go.mod` | Go | Go |
| `pyproject.toml` | Python | Python |

## P4. Setting Up a New Project

### Via Dashboard (recommended)

1. Navigate to `/setup` (or click "Setup Project" on the empty Home page)
2. Enter the absolute path to your game repo
3. Click **"Detect & Init"** — harness auto-detects the engine and scaffolds `.ludus/`
4. Review the generated config, edit name/description/conventions if needed
5. Set `PROJECT_REPO_LOCAL_PATH=/your/path` in harness `.env`
6. Click **"Reload & Go to Dashboard"** — the project badge lights up

### Via API

```bash
# 1. Scaffold .ludus/
curl -X POST http://localhost:3001/api/project/init \
  -H 'Content-Type: application/json' \
  -d '{ "path": "/path/to/your-game" }'

# 2. Set env + reload
echo 'PROJECT_REPO_LOCAL_PATH=/path/to/your-game' >> .env
curl -X POST http://localhost:3001/api/project/reload
```

### What init scaffolds

```
your-game/
├── .ludus/
│   ├── project.yaml      ← generated from auto-detection
│   └── rooms/             ← empty dir for Feature Rooms
└── prd/
    └── README.md          ← template explaining PRD format
```

## P5. Connecting an Existing Project

If the game repo already has `.ludus/project.yaml` (e.g., from a previous init or manual creation):

1. Set `PROJECT_REPO_LOCAL_PATH=/path/to/existing-repo` in harness `.env`
2. Restart the server or `POST /api/project/reload`
3. Dashboard immediately shows the project badge

The init endpoint refuses to run if `.ludus/project.yaml` already exists — use `/reload` instead.

## P6. PRD Documents

Product Requirement Documents live at **`game-repo/prd/`** (root level, not inside `.ludus/`), because PRD is product documentation, not harness configuration.

Format: Markdown, one file per major feature area. Example:

```
prd/
├── 01-core-gameplay.md
├── 02-combat-system.md
├── 03-economy.md
└── 04-progression.md
```

The Orchestrator agent reads these at cycle start when planning tasks. The `prd_path` field in `project.yaml` lets you override the default `prd/` location.

## P7. Milestone Lifecycle

Milestones live in **Mongo only** — no yaml files. The lifecycle:

```
1. Orchestrator reads PRD → proposes milestones
   → creates MilestoneModel with status: 'proposed', source: 'orchestrator'

2. Milestone appears in Inbox as 'milestone_proposal'
   → human confirms or rejects from Inbox (or /milestones page)

3. Confirmed → status: 'planned'
   → Orchestrator picks planned milestones for cycle goals

4. Cycle running → status: 'active'

5. All goals met → status: 'completed'
```

Humans can also create milestones directly from the `/milestones` page ("New Milestone" button).

Dashboard `/milestones` is a **management UI**: create, edit, reorder, confirm proposed, archive.

## P8. Feature Rooms (Project Side)

The Curator agent writes knowledge specs into `.ludus/rooms/`. At startup, `seedRooms()` scans both:

1. `harness-system/rooms/` — harness internal rooms (cycle engine, agent system, etc.)
2. `$PROJECT_REPO_LOCAL_PATH/.ludus/rooms/` — project Feature Rooms

Both sets are merged into the same Mongo Room tree.

**ID convention:** Project rooms should use a `p-` prefix (e.g., `p-farming`, `p-combat`) to avoid collision with harness room IDs (which are numeric like `01-cycle-engine`).

Room structure follows the same yaml format as harness rooms:

```
.ludus/rooms/
├── _tree.yaml          ← optional: defines room hierarchy
├── p-farming/
│   ├── room.yaml
│   └── specs/
│       ├── constraint-max-plots-6.yaml
│       └── decision-growth-stages.yaml
└── p-combat/
    ├── room.yaml
    └── specs/
```

If no `_tree.yaml` exists, each subdirectory with a `room.yaml` becomes a flat top-level room.

## P9. Agent Prompt Injection

Ludus owns all agent prompts (`harness-system/agents/*.md`). These are **generic stubs** that describe each role's responsibilities, output format, and rules without referencing any specific tech stack.

At spawn time, `buildProjectContextSection()` reads `getProjectConfig()` and appends a `## Project Context` section to every prompt:

```markdown
## Project Context
- **Project:** Zombie Farm — A zombie farming simulation
- **Engine:** Godot 4.6.1
- **Language:** GDScript
- **Test Runner:** GUT
- **Source dir:** scripts/
- **Tests dir:** tests/
- **PRD docs:** prd/

### Conventions
Use static typing (:=, -> void) throughout.
File names use snake_case.
...
```

When no project is loaded, the section is omitted — the generic stub stands alone.

**There is no prompt override mechanism.** The harness fully owns the prompt; the project only influences it through `project.yaml` fields.

## P10. First-Cycle Bootstrap

When a freshly initialized project has no milestones, no rooms, and minimal PRD:

1. User creates a cycle with a goal like "Read PRD and propose milestones"
2. Orchestrator reads `prd/` docs, decomposes the product into milestones
3. Each proposed milestone appears as an Inbox item
4. Human confirms/rejects milestones from Inbox or `/milestones` page
5. Confirmed milestones become the roadmap
6. Subsequent cycles target specific milestones: "Implement M0: Player Movement"

**Minimal bootstrap requires:** at least one PRD doc in `prd/` describing what the product should do. Without PRD, the Orchestrator has no product context and the plan will be generic.

## P11. Migration from Old `.ludus/` Format

Projects created before the Phase G rework may have:

```
.ludus/
├── agents/         ← DELETE (harness owns prompts now)
├── knowledge/      ← DELETE (move content to rooms/ as specs)
├── milestones/     ← DELETE (use dashboard to create milestones)
├── assets-planned.json  ← DELETE (scanner reads filesystem)
├── project.yaml    ← KEEP (update schema: remove paths.agents etc.)
└── rooms/          ← KEEP
```

Steps:
1. Delete `.ludus/agents/`, `.ludus/knowledge/`, `.ludus/milestones/`, `.ludus/assets-planned.json`
2. Edit `project.yaml`: remove `paths.agents`, `paths.knowledge`, `paths.milestones`, `paths.assets_registry`. Add `conventions:` field with the content from `knowledge/conventions.md`
3. Convert `knowledge/boot.md` content into a context spec in `.ludus/rooms/`
4. Convert `knowledge/glossary.md` into convention specs
5. Restart the server — it reads the new structure automatically
