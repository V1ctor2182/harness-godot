## 我的思考和问题

### Docker 为什么卡机
- 机器 16GB RAM，Docker 只分了 7.6GB
- 3 个 agent × 4GB = 12GB，光 agent 就超了 Docker 总内存
- 改了 container memory 4096→2048，agent slots 3→2，infra slots 8→3
- Docker Desktop 内存调到 8GB（不是 12GB — 要给 macOS + 浏览器 + IDE 留空间）

### knowledge files 和 Room Specs 什么区别
- knowledge/ 是旧系统：flat markdown 文件，全局 top-15 注入，无归属
- Room/Spec 是目标系统：树形结构，按 task 精准匹配，typed specs + 继承
- techdesign 应该只描述目标系统，清理了所有 legacy 引用

### bootstrap 文件 (boot.md, conventions.md, glossary.md) 为什么不放进 Room
- 问了为什么这三个文件单独存在而不是 Room 里的 specs
- 结论：应该放进 00-project-room 作为 specs，通过继承所有 agent 都能拿到
- 已更新文档，去掉独立的 knowledge/ 目录概念

### Context Builder 是什么
- 不是 agent，是 Server 端的确定性模块
- 每次 spawn agent 前执行：Task→Room 匹配 → 收集 specs + 继承 → 排序截断 → 拼 prompt
- 反馈闭环：agent 说哪些 spec 有用/没用 → qualityScore 更新 → 下次排序变化

### Plan 阶段加 Reviewer
- Orchestrator 的 plan 只有格式校验 (plan-validator)，没有质量审查
- 加了 Reviewer plan-review mode，不是新 agent，是复用 Reviewer
- mode 由 Context Builder + Spawner 决定（注入不同 Room specs），reviewer.md 不变

### Questions 和 Retry 是两回事
- 一开始搞混了：把 Orchestrator 问人问题当成了 retry
- 实际上：questions 是 plan 生成前的正常协商（不算 retry），Request Changes 才算 retry
- 两个人工介入点完全分开：Q&A 在 validator 之前，Human Review 在 reviewer 之后

### 审批 pending 会不会卡死
- pending job 永远等，没有超时
- 电脑关了重启后：pending job 还在 DB，不丢数据，但正在跑的 agent run 丢失（浪费 cost）
- Discord 通知设计标为 future feature
